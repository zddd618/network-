# 前端展示功能分析与解决思路

## 1. 比赛要求映射

根据 B-EP1 赛项指南，作品展示页需要服务于“智能体互联网应用系统”的评审，而不是普通项目介绍页。页面应覆盖以下要求：

- 前瞻性与融合度：突出生成式 AI、大模型、智能体互联网、多智能体协同与真实图书配送痛点。
- 技术创新性与可行性：清楚呈现 ReAct 推理-行动-观察循环、提示词策略、工具函数接口、任务分配与路径规划。
- 系统可用性与文档规范性：提供直观、交互友好、可演示的 GUI，能支撑设计文档和演示视频。
- 交付物标准：页面可作为在线演示平台入口，并能链接源代码、设计文档、演示视频与部署说明。

## 2. 展示页面核心目标

网页应让评委在 3-5 分钟内看懂四件事：

1. 项目解决什么问题：非专业用户用自然语言指挥多台机器人完成复杂配送。
2. 为什么属于智能体互联网：系统通过标准化工具函数、状态反馈、任务协同和语义路由连接人、LLM 智能体与机器人智能体。
3. 技术方案是否可行：ReAct 负责意图澄清与函数调用，A* 负责路径代价评估，最小代价策略负责任务分配，CBS 负责无冲突路径规划。
4. 效果是否可信：展示模拟数据、真实图书馆部署、4 台 AGV、任务成功率等证据。

## 3. 必须实现的展示功能

### 3.1 首屏项目总览

- 项目名称：基于 ReAct 模式的大模型驱动多移动机器人协作系统。
- 核心一句话：通过多轮对话理解模糊指令，自动规划并调度多台 AGV 完成图书配送。
- 关键指标：总体成功率 74.8%、多步多函数成功率 96.1%、无函数识别成功率 96.7%、实机 4 台 AGV 部署。
- 主操作：开始演示、查看架构、查看实验结果。

### 3.2 交互式任务演示

展示一个完整的 ReAct 链路：

- 用户输入模糊任务，例如“找 2 本热门科幻书送到 1 号桌”。
- 系统判断信息是否完整，不完整则主动澄清。
- 意图明确后生成工具函数调用序列。
- 页面展示每一步 Thought、Action、Observation。
- 执行中可模拟机器人状态变化或配送失败，触发 revise 重规划。

建议前端表现：

- 左侧为对话窗口。
- 中间为 ReAct 时间线。
- 右侧为机器人状态、任务队列和路径地图。

### 3.3 系统架构展示

页面应展示从上到下的系统链路：

- 用户自然语言输入。
- LLM/ReAct 任务规划智能体。
- 工具函数库：search_books、recommend_similar_books、get_target_location、get_robots_state、calculate_optimal_allocation、send_agvs_task_workflow。
- 任务分配层：基于 A* 计算路径代价，求解最小总代价匹配。
- 调度执行层：CBS 集中式无冲突路径规划。
- AGV 机器人端：执行、状态上报、异常反馈。

### 3.4 多机器人地图与状态看板

需要展示系统“可用”而非只讲概念：

- 地图区域：图书架、服务台、目标桌位、机器人位置、规划路径。
- 机器人状态：空闲、前往书架、取书、配送中、异常、完成。
- 任务队列：每个目标图书、目标位置、分配机器人、当前进度。
- 冲突规避：用时间线或路径颜色展示 CBS 规避冲突。

早期可以用模拟数据实现，后续接入真实机器人状态 API。

### 3.5 技术创新与评分点展示

建议按赛项评分维度组织：

- 创意：自然语言驱动图书配送，把智能体互联网落到真实机器人协作。
- 方案实现：前端 GUI、智能体规划、工具函数、机器人调度系统闭环。
- 技术应用：LLM、ReAct、多智能体协同、A*、CBS、结构化 JSON 函数调用。
- 编程开发：前后端模块化、状态管理、接口协议、部署方式。
- 设计：交互清晰、演示链路完整、答辩友好。
- 效果：模拟数据与实机部署结果。

### 3.6 实验与应用效果展示

建议使用指标卡、表格和图表：

- 自建模拟数据集 900 条。
- multi_function：510 条，成功率 70.0%。
- single_function：156 条，成功率 58.3%。
- multi_step_multi_function：204 条，成功率 96.1%。
- no_function：30 条，成功率 96.7%。
- 总体成功率：74.8%。
- 实机场景：图书馆，4 台 AGV，多机器人图书配送。

## 4. 推荐技术方案

### 4.1 第一阶段：静态可演示页面

目标是在短时间内形成比赛可展示版本：

- 使用 HTML、CSS、JavaScript 构建单页应用。
- 内置模拟任务、机器人状态与 ReAct 步骤。
- 不依赖网络和 npm 安装，打开即用。
- 适合录制演示视频和现场答辩。

### 4.2 第二阶段：接入真实接口

当前端原型稳定后，逐步接入后端：

- `POST /api/intent`：提交用户自然语言指令。
- `POST /api/react/step`：返回下一步 Thought/Action/Observation。
- `GET /api/robots`：获取机器人状态。
- `GET /api/tasks`：获取任务队列。
- `GET /api/map`：获取地图节点、路径与障碍物。
- `POST /api/tasks/replan`：触发反馈重规划。

### 4.3 第三阶段：答辩增强

- 增加“评审模式”：自动播放完整任务流程。
- 增加“技术模式”：显示 JSON 函数调用、接口协议和算法解释。
- 增加“部署模式”：展示实机图片、视频、硬件清单和运行环境。

## 5. 页面结构建议

推荐单页滚动结构：

1. Hero：项目名、核心价值、关键指标、演示入口。
2. Live Demo：自然语言输入、ReAct 时间线、机器人状态地图。
3. Architecture：系统架构图和模块说明。
4. Algorithm：任务分配、A* 代价评估、CBS 路径规划。
5. Results：模拟实验和实机部署效果。
6. Innovation：三到五项创新特色。
7. Deliverables：设计文档、源码、演示视频、在线部署链接。

## 6. 视觉与交互方向

- 风格：科技感但克制，突出工程可信度。
- 配色：深色控制台背景 + 高对比状态色，不做单一蓝紫渐变。
- 组件：地图、状态条、时间线、指标卡、函数调用面板。
- 动效：机器人移动、步骤点亮、任务状态变更，避免过度装饰。
- 响应式：答辩大屏优先，同时兼容笔记本和移动端查看。

## 7. 风险与注意事项

- 不要只做宣传页，必须有可交互演示。
- 不要隐藏技术细节，评委需要看到函数调用、任务分配和反馈闭环。
- 不要把实验指标写得过度绝对，应注明来自自建模拟数据集与真实场景部署验证。
- 匿名要求下，页面和提交资料中不要出现学校、团队、队员身份信息。
